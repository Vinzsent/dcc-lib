## Fix: Employee RFID tap "Network error"

### Confirmed root cause
The live `employees` table is missing columns (`eid`, `campus`, ...) that the migration defines and `ScannerController::scanEmployee()` queries against. Each employee tap throws `SQLSTATE[42S22]: 1054 Unknown column 'eid'` → 500 HTML response → JS shows "Network error".

### Execution steps

**Step 1 — Inspect the actual live schema (read-only).**
Run `DESCRIBE employees; DESCRIBE employee_logs;` (via `mysql` CLI or `php artisan tinker` with `Schema::getColumnListing()`) to get the *exact* columns that currently exist. This is essential — I won't write the corrective migration blind. I'll compare the live columns against what the code needs.

**Step 2 — Create a corrective migration.**
Generate a new migration (e.g. `2026_07_08_fix_employees_missing_columns`) that `ALTER`s the `employees` table to add every missing column the code expects, matching the existing migration definition:
- `eid` (string, unique, nullable for backfill safety)
- `campus` (string, nullable)
- `rfid`, `firstname`, `middlename`, `lastname`, `department`, `position`, `employment_type` (only those confirmed missing in step 1)
- Each `ALTER` will use `Schema::hasColumn(...)` guards so it's idempotent and safe to re-run.

If `employee_logs` is also missing columns (it's queried in the same flow), apply the same treatment there.

**Step 3 — Run the migration.**
`php artisan migrate`

**Step 4 — Data check (important).**
After adding `eid` (NULL for existing rows), verify the employee being tapped can still be matched. `scanEmployee()` queries `WHERE eid = ? OR rfid = ?`, so as long as the tapped value is stored in the `rfid` column, the lookup will work even with NULL `eid`. If step 1 reveals the employee ID is stored under a *different* column name, I'll flag it and we'll discuss a backfill before proceeding.

**Step 5 — Verify the fix.**
- Tap the employee RFID again → scanner should show "Time In / Time Out".
- Confirm `storage/logs/laravel.log` shows no new `Unknown column` errors.
- Smoke-test the admin employee pages (the `campus`-filter query that errored at 09:20:51).

### What I will NOT change
- No edits to `ScannerController`, models, or the blade — the code is correct; the database schema is the problem.
- No re-running/deleting the original migration.
- No touching the working student/inouts tables.

### Notes
- This is a schema change to a table with existing data. The `Schema::hasColumn` guards and nullable columns make it low-risk, but I'll show you the migration content and the `DESCRIBE` diff before running it.
